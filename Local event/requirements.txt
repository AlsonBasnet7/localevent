Group Assignment Guidelines and Specifications
Group Website Design Project
Assessment Overview (Group Work – 60 Marks Total)
In this assessment, students will design, develop, and document a fully functional multi-page website using HTML5 and CSS — without automated code generators or CSS frameworks. Each team will simulate a real-world client engagement: gathering requirements from a mock client, producing professional design and technical documentation, and delivering a validated, accessible website prototype.
Business Scenario Selection
Each group will choose one business category from the list below and build a website for a fictional business of that type. This simulates real professional practice, a web designer does not always get to choose their client, but they do bring creative judgment to every project.
Every group must select a topic from listed below. Register your choice with your lecturer using the scenario registration process described below.

Learning Outcomes Addressed
By completing this assessment, students will be able to:
Apply web-related programming skills to design and develop a web-based system for a particular business scenario. (ULO 3)
Understand and evaluate business problems related to web design and suggest possible solutions. (ULO 4)
Work autonomously as well as within a group to develop a solution to a business scenario and communicate this solution to a diverse range of stakeholders. (ULO 5)
Assess the ICT profession and the expectations of ICT professionals in web design roles. (ULO 6)
Project Description
Based on your registered business scenario, your group will build a website consisting of at least four interlinked pages. The specific required pages for each scenario are listed in the Scenario Selection table above.
Across all scenarios, the website must:
Use semantic HTML5 elements (header, nav, main, section, article, footer) — no inline styles, no CSS frameworks such as Bootstrap or Tailwind.
Use a single external CSS file (styles.css) for all visual styling.
Include consistent navigation across all pages with working internal links.
Demonstrate effective typography, colour scheme, and appropriate use of images aligned with the chosen business brand.
Be responsive — pages must display correctly on both desktop and mobile screen widths.
Pass W3C HTML and CSS validation with no critical errors.
Include a contact or enquiry form (name, email, message fields minimum) on the appropriate page.
Important Restrictions
Do not use CSS frameworks (Bootstrap, Tailwind, Foundation, etc.) or website builders (Wix, Squarespace, WordPress, etc.).
Do not use AI code-generation tools to write your HTML or CSS.
All images must be royalty-free (e.g., Unsplash, Pexels) or original. Cite all image sources in your report.

2. Design and Technical Documentation (30 marks)
Each group must submit a report of 1,200–1,800 words including all of the following sections. Each section must be clearly labelled with a heading.
Section
Required Content
Client Requirements Summary
A written record of your group's mock-client consultation process. Describe how your group identified the client's needs (e.g., a written questionnaire, mock meeting notes, or an email exchange). List at least six specific website requirements that emerged from this process and explain how each shaped your design or technical decisions.
Site Map and Wireframes
A site map diagram showing the page hierarchy and navigation structure of your website. Wireframes for at least two pages (hand-drawn or created digitally) showing layout before coding began — see the Wireframe Format Reference on the following page for the expected level of detail. Briefly explain the design rationale behind your major layout decisions.
Style Guide
Document your visual design decisions: colour palette with hex codes, font names and size hierarchy, spacing and layout conventions, and image or icon usage guidelines. Explain how these choices reflect your chosen business brand and target audience.
Technical Implementation
Explain your HTML5 structure (semantic elements used and why), your CSS approach (key selectors, box model, how you achieved responsiveness), any significant challenges encountered, and how your group resolved them. Include screenshots of your W3C HTML and CSS validator results showing your final validation status.
Individual Reflection (one per member)
Each group member writes approximately 100 words describing their specific contribution to the project, one thing they learned about web design or working in a team, and one thing they would do differently if repeating the project.

What Your Wireframes Must Show
Element
What to Include
Page zones / sections
Clearly labelled rectangles for each zone: header, navigation, hero, main content, sidebar if used, footer.
Navigation structure
A labelled navigation bar showing the page names — does not need to show final styling, just position and labels.
Content placeholders
Use placeholder labels such as [Headline Text], [Image 300×200], [Paragraph content], [Button — Book Now] to indicate what goes where without designing the actual content.
Proportions
Drawn roughly to proportion — if the image is twice the width of the text column, the wireframe should show that relationship.
Annotations
Short notes beside elements explaining design decisions, e.g. 'Nav sticks to top on scroll' or 'Two-column layout collapses to one column on mobile'.
Page label
Each wireframe must be clearly labelled with the page name at the top, e.g. 'Home Page — Desktop View'.
Wireframes may be hand-drawn and photographed, or created digitally using any tool (pen and paper, PowerPoint, draw.io, Figma, etc.). They must be embedded or pasted into your Word report document, do not submit them as separate files.
